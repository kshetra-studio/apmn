Google ADK output structure — pros & cons
Flat Agent (default)
Simplest generated code
Gemini handles sequencing naturally
Easy to read and modify
Order not structurally enforced
Parallel branches not explicit
No sub-agent isolation
SequentialAgent
Mirrors BPMN flow order exactly
Execution order guaranteed
Better for regulated workflows
More generated code complexity
Harder for non-engineers to follow
ADK SDK still maturing
Nested (Parallel+Sequential)
Faithfully maps BPMN topology
FORK_JOIN → ParallelAgent
Clean sub-agent boundaries
Complex generated code
Harder to debug
Brittle if BPMN changes often
Recommendation — config flag with default
Add adk_style: flat | sequential | nested to YAML config. Default = flat.
For Orkes sub-processes: add subprocess_handling: flatten | subworkflow. Default = flatten.
Orkes sub-process handling
Flatten (default)
Single flat task list
Simple to register in Orkes
Loses subprocess boundary
Sub-workflow
Separate reusable workflow
Matches Orkes SUB_WORKFLOW task
Multiple JSON files to register
Inline DO_WHILE
Loop subprocess → DO_WHILE
Only works for looping patterns
Edge case — low priority
Post-transform: guide vs sub-tool
POST_TRANSFORM.md (inline guide)
Zero extra tooling, ships with output
Human-readable, auditable
Works offline
User must act on it manually
Stale if BPMN is re-generated
twintrack adjust sub-tool
Interactive — patch specific touchpoints
Always current (reads live output)
Can apply changes in one command
More build effort (v0.3 scope)
Needs output files to exist first
Recommendation — both, in phases
v0.2: Generate POST_TRANSFORM.md alongside every output.
v0.3+: twintrack adjust — reads files, prompts per touchpoint, patches in-place.

# ISSUE: switchCaseValue is a stub — wire to your worker output.

# ACTION: change inputParameters.switchCaseValue to
# ${apply_clinical_admission_criteria.output.approved}

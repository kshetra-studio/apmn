Existing alternatives — why none fill the gap
Process / Decision Standards
  • BPMN 2.0 — deterministic, no AI concepts
  • CMMN — adaptive cases, but no agents
  • DMN — decision tables, not LLM-aware
  • BPEL — XML orchestration, pre-AI era
  • UML Activity — too technical, no semantics
  • Petri Nets — academic, not business-readable
AI Orchestration Tools (proprietary)
  • LangGraph — code-first, Python only
  • CrewAI — role-based, no visual notation
  • AutoGen — conversation-based, MS-specific
  • Dify / Flowise — visual but proprietary
  • AWS Step Functions — AWS-locked JSON
  • Bedrock Flows — AWS-only, limited
  • Vertex AI Pipelines — ML only, not business
What's missing in ALL of them
  • No open, portable notation standard
  • No business-analyst-readable visual spec
  • No first-class: agents, RAG, MCP, confidence
  • No probabilistic routing (confidence gates)
  • None compile to multiple targets
  • No standard for multi-agent handoffs
  • None bridge BA ↔ engineer in one diagram
The gap is real and unoccupied
There is no open, vendor-neutral, visual notation standard for designing AI agent workflows that both business analysts and engineers can use. The market gap is: BPMN's accessibility + AI-native semantics + open standard + multi-target compilation.
New task node types — first-class AI citizens
🤖 agentTask
LLM-driven task. Attrs: model, system_prompt, temperature, tools[]
<apn:agentTask model="gemini-2.0-flash">
📚 ragTask
Retrieval-augmented generation. Attrs: vector_store, top_k, similarity_threshold
<apn:ragTask store="patient_docs">
🔌 mcpToolTask
Invoke a specific MCP server tool. Attrs: server, tool_name, input_schema
<apn:mcpToolTask server="fs" tool="read">
👤 humanInLoopTask
Agent proposes, human approves. Attrs: confidence_threshold, timeout, escalate_to
<apn:humanInLoopTask threshold="0.9">
↗ agentHandoff
Delegate to specialist agent. Multi-agent orchestration. Attrs: target_agent, context
<apn:agentHandoff target="billing_agent">
🧠 memoryTask
Read/write agent memory (session, long-term). Attrs: store, operation, key
<apn:memoryTask op="write" store="session">
⬡ vectorTask
Embed content into / query vector store. Attrs: store, operation, model
<apn:vectorTask op="embed" store="kb">
📡 observeEvent
Emit LLM trace/span to observability platform (LangFuse, Phoenix, OTEL)
<apn:observeEvent platform="langfuse">
New gate types — probabilistic and AI-native routing
◇ confidenceGate
Routes based on agent confidence score (0.0–1.0). High → proceed. Medium → human review. Low → retry or escalate. Replaces binary exclusiveGateway.
◇ reasoningGate
Routes by parsing LLM chain-of-thought output. The model's own reasoning determines the path — no hard-coded condition needed.
◇ modelVersionGate
A/B or canary route between model versions. Traffic split by percentage. Ideal for safe model upgrades.
◇ semanticGate
Routes by vector similarity — input is embedded and matched against intent clusters. Most semantically similar branch is taken.
◇ mcpGate
Calls an MCP tool and routes on result. E.g. compliance-check MCP → pass/fail/review. First-class MCP in the flow.
◇ escapeGate
Auto-escalation. If any upstream agent fails, times out, or hits a confidence floor — route to human or fallback agent.
Extension of BPMN 2.0 vs a new notation
Option A — Extend BPMN 2.0
Use <bpmn:extensionElements> with a custom apn: namespace.

Pro: Backward-compatible. Huge existing BPMN tooling ecosystem.
Pro: Business analysts already know BPMN shapes.

Con: Constrained by BPMN's deterministic mental model.
Con: Can't add genuinely new gate shapes without forking BPMN.io.
Con: You'll always be "BPMN with extensions", not a distinct standard.
Option B — New notation: APN
AI Process Model and Notation — new open standard. YAML serialisation. Visual spec. Apache 2.0. TwinTrack is the reference compiler.

Pro: Own the standard, not an extension of someone else's.
Pro: Design the right mental model (probabilistic, not deterministic).
Pro: First mover — no competitor exists today.

Con: New visual language = learning curve for BAs.
Con: Need to build or fork modeller.
Con: Harder to get enterprise adoption without existing tooling.
Recommendation — Hybrid: APN as a superset of BPMN 2.0
Start with BPMN 2.0 as the base serialisation. Every APN diagram IS a valid BPMN 2.0 file. APN-specific nodes use extension elements. The APN spec defines the extensions formally (XSD + visual spec).

Publish at spec.apmn.kshetra.studio under Apache 2.0. TwinTrack is the reference parser and compiler. APMN is the source language. TwinTrack is the compiler.
What an APMN file looks like — patient admission example
# APMN — AI Process Model and Notation v0.1
# xmlns:apmn="http://apmn.kshetra.studio/ns/1.0"

process:
  id: patient_preadmission
  name: Patient Preadmission
  targets: [orkes, google_adk, langraph]

nodes:
  - id: verify_insurance
    type: agentTask # NEW — first-class agent
    model: gemini-2.0-flash
    tools: [mcp://insurance-api, mcp://medicare-lookup]
    confidence_threshold: 0.95

  - id: gw_confidence
    type: confidenceGate # NEW — probabilistic routing
    source: verify_insurance.confidence
    routes:
      high: ">= 0.95 → proceed"
      medium: "0.7–0.95 → human_review"
      low: "< 0.7 → escalate"

  - id: notify_compliance
    type: mcpToolTask # NEW — MCP as first-class node
    server: mcp://compliance-checker.kshetra.studio
    tool: check_hipaa_consent
Road to becoming an open standard
Phase 1 — Now APN in TwinTrack
  • APMN YAML schema v0.1 published
  • TwinTrack reads APMN YAML natively
  • BPMN 2.0 + apn: extensions as alt input
  • Apache 2.0, GitHub: kshetra-studio/apmn
  • New node types: agentTask, ragTask, mcpToolTask, confidenceGate
Phase 2 — v1.0 Tooling + Community
  • Visual spec — shape library for each node type
  • BPMN.io plugin to render APN nodes
  • VS Code extension — APN language support
  • APMN Playground — paste YAML, see diagram
  • Submit to Orkes, Google as adopting parties
Phase 3 — v2.0 Standard + Ecosystem
  • Submit APN to OMG as BPMN 3.0 input
  • Partner with Camunda / Flowable communities
  • APMN Working Group — open governance
  • Certification: "APMN-compatible" badge
  • Multiple reference implementations (JS, Java, Go)
Strategic position
TwinTrack is the compiler. APMN is the source language. Just as TypeScript compiles to JavaScript for multiple targets, APN compiles to Orkes, Google ADK, LangGraph, AWS Bedrock. The standard lives independently of any runtime vendor — that is the moat.
